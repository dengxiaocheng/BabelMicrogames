# Windows 本地运行脚本

本文描述 Windows 本地机器上的运行脚本如何归档、交付和维护。

## 目标

Windows 本地脚本应像 Termux 运维脚本一样，有固定归档位置和固定文档入口。

这样做是为了避免：

- 脚本散落在聊天里
- 本地机器上只剩手工复制版本
- 同一类入口在多个目录重复漂移
- 之后追溯“哪个才是当前标准入口”时只能翻聊天记录

## 归档规则

Windows 本地可复用脚本统一归档到：

- `scripts/windows/`

对应运维说明统一归档到：

- `docs/operations/WINDOWS_LOCAL.md`

不要把 Windows 本地运行脚本直接散落在：

- 仓库根目录
- 临时附件说明里
- 只有聊天消息、没有仓库文件的片段里

## 适用范围

以下内容默认都按 Windows 本地运行脚本处理：

- 本地启动器
- 本地 resume / reconnect 入口
- 本地重写脚本
- 本地检查和排障脚本
- 本地一键执行脚本
- 需要用户多次执行的复制型命令集合

## 交付规则

默认规则：

- 先把 canonical 脚本提交到 `scripts/windows/`
- 再告诉用户执行哪个稳定路径
- 如果需要额外解释，再把说明写进本文档，而不是堆进脚本正文

如果用户明确说明当前 Windows 机器不方便直接从仓库取文件，或复制行为容易损坏格式，则：

- 仍然先把 canonical 脚本提交到 `scripts/windows/`
- 然后再在聊天里提供完整可手工复制版本

也就是说：

- canonical source 仍然在仓库里
- 但交付层可以补一份可直接复制的文本

## 与 Termux 的关系

这套规则和 Termux 一致，但不混目录：

- Termux 本地脚本继续放 `scripts/` 下的 Termux 入口
- Windows 本地脚本统一放 `scripts/windows/`

两者共同遵守的原则是：

- 先落仓库
- 再交付用户
- 文档统一收口到 `docs/operations/`

## 后续约定

以后只要出现新的 Windows 本地运行入口，就应同时检查：

- 是否应该新增 `scripts/windows/...`
- 是否需要在本文补充对应说明
- 是否需要在 `README.md` 或命令清单中登记
