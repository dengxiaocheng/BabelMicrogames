# PLAN_GLM51_BABEL_MICROGAME_FACTORY

## 背景

- 接下来约两个月内，有大量 `ClaudeCode + GLM 5.1` token 可用，量级为几十亿。
- 当前已知限制是：ClaudeCode 在大项目上容易崩，尤其当总代码规模明显超过 `5000` 行时，人工干预成本会迅速上升。
- 目标不是让它硬写 Babel 主体，而是让它高频稳定地产出“Babel 相关的某一条创意线”的小型游戏或垂直切片。

## 核心判断

不要让 ClaudeCode 负责“大项目”。

应该让它负责：

- 小游戏
- 单创意线切片
- 固定模板上的局部实现
- 严格拆分后的低耦合任务

应该由我这边负责：

- 计划
- 拆任务
- 边界控制
- 验收
- 集成

也就是说，正确用法不是“让 ClaudeCode 自动做 Babel”，而是：

- 先做一套 `Babel 微游戏工厂`
- 再让 ClaudeCode 在这套工厂里批量生产小而稳的可玩原型

## 总目标

在两个月内建立一条稳定流程：

1. 从一条 Babel 创意线快速生成小游戏 brief
2. 自动拆成 ClaudeCode 能稳定完成的小任务
3. 用固定模板和固定边界驱动 ClaudeCode 编码
4. 产出一批 Babel 相关小游戏或垂直切片
5. 把可复用部分反向沉淀成下一轮模板

## 当前落地状态

目前第一版工厂脚本已经落地：

- `scripts/microgame_factory_create.sh`
- `scripts/microgame_factory_start.sh`

它当前能做的事：

1. 生成一条创意线对应的 `creative card`
2. 生成一页 `mini GDD`
3. 生成标准 `TASKS.md`
4. 自动注册 6 个默认 worker packet：
   - `foundation`
   - `state`
   - `content`
   - `ui`
   - `integration`
   - `qa`

并且现在已经有一键开始入口：

- `sh scripts/microgame_factory_start.sh --preset peigei`

它会直接：

1. 生成工厂产物
2. 注册默认 worker packets
3. 以 `run-once` 无人值守模式立刻派出当前第一个 Claude worker

而且现在开始脚本会自动用当前项目的 `task_prefix` 做过滤，只派这条创意线自己的 worker，不会被历史 smoke 或别的项目队列串进来。

`run-once` 使用 `claude -p` 执行 packet，避免交互式 trust prompt 卡住不可接回的 TTY。需要人工进入 ClaudeCode 时，可给开始脚本加 `--interactive`。

这意味着工厂已经不再只是计划文档，而是有了第一版实际生成器。

## 目标产物

这套计划最终不只要有游戏，还要有“产线”。

产物分三层：

### 1. 工厂层

- 微游戏模板
- 任务拆分标准
- ClaudeCode 执行提示模板
- 自动审查/验收清单
- 项目归档规范

### 2. 游戏层

- 一批 Babel 相关小游戏
- 一批 Babel 相关创意切片
- 每个项目都保持在 ClaudeCode 可控范围内

### 3. 资产层

- 可复用 UI/状态/输入模板
- 可复用文本风格包
- 可复用 Babel 世界观约束包
- 可复用玩法骨架

## 基本策略

### 策略一：只做“小而完整”的项目

不要试图让 ClaudeCode 一次性写完整大作。

优先目标是：

- `1` 个玩法核心
- `1` 个主循环
- `1` 套状态系统
- `1` 个小范围内容池

每个项目尽量控制在：

- 总代码规模 `<= 3000 ~ 5000` 行
- ClaudeCode 单轮修改规模 `<= 300 ~ 500` 行

### 策略二：固定底座，变化内容

ClaudeCode 最适合在固定模板上做局部生成，而不适合频繁推翻架构。

所以要先固定：

- 技术栈
- 目录结构
- 启动方式
- 资源组织方式
- 状态对象结构
- UI 骨架

之后只让它变化：

- 创意线
- 内容
- 小玩法规则
- 场景脚本
- 数值和文本

### 策略三：让 Codex 做“总控”，ClaudeCode 做“工人”

角色分工应固定：

- 你：选题、验收、决定是否继续扩线
- Codex：计划、拆任务、审查、整合、重构、把关
- ClaudeCode(GLM 5.1)：执行被约束的小任务

也就是说：

- ClaudeCode 不负责决定架构
- ClaudeCode 不负责长期项目治理
- ClaudeCode 只负责在已定义边界内产出代码

### 策略四：先做“微游戏工厂”，再做“微游戏”

前两周的重点不应该是拼命出游戏，而是先把产线搭起来。

否则后面几十亿 token 只会被浪费在：

- 反复崩项目
- 人工修边界
- 人工重写目录
- 人工救火

## 推荐产出形态

不要以“一个巨大 Babel 衍生项目”为目标。

更适合的产出形态是：

### A. 微游戏

例子：

- 搬砖节奏游戏
- 配给分发小游戏
- 工头巡视规避
- 祈祷/献祭选择小游戏
- 休息区社交事件小游戏

特点：

- 小
- 快
- 可玩
- 可重复生成

### B. 创意切片

例子：

- “塔底工人第一天”
- “饥饿与配给”
- “夜间祈祷”
- “工头点名”
- “工地事故”

特点：

- 强主题
- 世界观浓度高
- 玩法不必太长
- 适合快速验证某条创意线

### C. 固定模板的变体

例如同一模板下不断换：

- 职责
- 阶层
- 资源压力
- 社交关系
- 日夜节奏

特点：

- 复用率高
- 便于 ClaudeCode 持续稳定输出

## 技术路线建议

### 第一优先：Web 微游戏模板

推荐优先做：

- 单页 HTML/CSS/JS
- 或极轻量 canvas / Phaser 风格模板

理由：

- 启动快
- 目录简单
- 依赖少
- 对 ClaudeCode 更稳
- 更适合控制在 `5000` 行以内

### 第二优先：Godot 小模板

如果一定要更靠近 Babel 正式方向，可以用：

- 固定 Godot mini-template
- 限制脚本数量
- 限制场景层级

但这条线应放在模板稳定之后，不宜一开始就作为主力批量产线。

### 不建议：直接拿大型 Babel 主项目当 ClaudeCode 工作区

理由：

- 上下文太重
- 变更面太大
- 很难让它稳定保持边界
- 人工 review 成本太高

## 微游戏工厂结构

建议分成四个固定步骤。

### Step 1：创意卡

每次先定义一张卡：

- 名称
- 核心情绪
- 核心玩法
- Babel 对应创意线
- 目标时长
- 失败条件
- 成功条件

这一步控制项目不要散。

### Step 2：Mini GDD

只写一页：

- 核心循环
- 状态变量
- 输入输出
- UI 组成
- 内容池

这一步防止 ClaudeCode 自行发散。

### Step 3：任务包

把项目拆成 5 到 12 个小任务。

每个任务要求：

- 只做一件事
- 最好只改 `1 ~ 3` 个文件
- 单次净新增尽量不超过 `300 ~ 500` 行
- 完成后必须可运行或可验证

### Step 4：执行与审查

循环固定为：

1. Codex 下发单任务
2. ClaudeCode 只做该任务
3. 本地运行/检查
4. Codex 审查并决定下一个任务

## 任务拆分标准

这是这套计划的核心。

### 强约束

- 不允许一个任务同时做架构 + 内容 + UI
- 不允许一个任务跨多个子系统大面积改动
- 不允许一个任务把项目规模直接推过可控阈值
- 不允许 ClaudeCode 自己新增“顺手的大重构”

### 推荐任务粒度

适合 ClaudeCode 的任务应该像这样：

- 建立状态对象
- 做主循环
- 接上输入
- 做一个 UI 面板
- 写一组事件数据
- 写一组数值规则
- 加一个结算页
- 修一个明确 bug

不适合的任务是：

- “把整个游戏做完”
- “顺手优化整个工程结构”
- “把这套玩法和另一个系统合并”

## ClaudeCode 操作方式

不要直接给它一句大话。

应该形成固定操作模板：

### 输入包

每轮只给：

- 当前任务说明
- 当前约束
- 必须修改的文件
- 禁止修改的文件
- 验收标准

### 输出要求

要求它：

- 只完成当前任务
- 不扩展范围
- 不改未授权结构
- 说明修改点

### 停止条件

每轮必须停在：

- 任务完成
- 或遇到阻塞

而不是继续“顺手往下写”。

## 两个月建议节奏

### 第 1 阶段：工厂搭建（第 1~2 周）

目标：

- 定模板
- 定技术栈
- 定拆分标准
- 定 prompt 模板
- 定审查清单

交付：

- `1` 套微游戏模板
- `1` 套任务拆分标准
- `1` 套 ClaudeCode 执行模板

### 第 2 阶段：首批验证（第 3~4 周）

目标：

- 先做 `2 ~ 3` 个小游戏
- 验证这条产线到底稳不稳

重点不是数量，而是：

- 平均人工干预次数
- 平均单项目完成时间
- 哪些任务最容易崩

### 第 3 阶段：批量扩线（第 5~8 周）

目标：

- 批量做 Babel 创意线小游戏
- 逐步积累模板和组件

建议目标：

- `6 ~ 12` 个可运行原型

## 成功标准

如果两个月后达到以下状态，就算这条计划成功：

- 已有一套稳定微游戏模板
- 已有一套固定的 ClaudeCode 拆任务方法
- 已有一批 Babel 创意线小游戏
- 每个新项目都不需要从零摸索流程
- ClaudeCode 可以在多数任务上稳定低干预产出

## 风险

### 风险一：模板不够硬

如果模板边界不硬，ClaudeCode 还是会乱改结构。

### 风险二：任务拆分过大

如果任务太大，还是会退回人工救火。

### 风险三：项目类型太复杂

如果一开始就选复杂 Godot 项目，产线很容易崩。

### 风险四：没有沉淀

如果每次都只做新游戏、不回收模板，token 会被不断重复浪费。

## 下一步建议

下一步不要直接开写游戏。

下一步应该先做这三样：

1. 定一个微游戏模板技术栈
2. 写一份 ClaudeCode 任务拆分标准
3. 选第一条 Babel 创意线，做第一个验证项目
